# ADR-0005: Хранение иерархии категорий

**Статус:** принято · **Дата:** 2026-05-23 · **Скоуп:** US-CAT-05

## Контекст
Категории — seed-данные, меняются редко, активно читаются (левое меню,
выбор категории, breadcrumbs). Хранение и алгоритм обхода живут в B2B;
B2C либо получает плоский список и собирает дерево у себя (что и
зафиксировано в каноне B2C-5), либо принимает уже собранные ответы
(breadcrumbs, детали категории). Этот ADR фиксирует ожидание B2C к
схеме хранения на стороне B2B.

## Рассмотренные подходы

1. **PostgreSQL `ltree`** — отдельный тип с GiST-индексом, путь хранится
   как `electronics.smartphones.android`. Breadcrumbs — один `WHERE path
   @> :node_path` за O(глубина); подразделы — `path <@ :ancestor_path`.
   Требует `CREATE EXTENSION ltree` и правил мэппинга slug → label.

2. **Adjacency list (`parent_id` FK) + рекурсивный CTE.** Самая простая
   таблица: одна колонка `parent_id` на строку. Breadcrumbs — рекурсивный
   `WITH RECURSIVE`. Поддерживается всеми Postgres-инсталляциями без
   расширений; orphan находится одним `LEFT JOIN parent ON child.parent_id
   = parent.id WHERE child.parent_id IS NOT NULL AND parent.id IS NULL`.

3. **Materialized path** (текстовая колонка с разделителями `/electronics/
   smartphones/android`). Чтение цепочки — один SUBSTRING/split, без
   рекурсии. Минусы — при переименовании любого предка нужно атомарно
   обновить путь всем потомкам, иначе расхождение между `parent_id` и
   `path`.

## Решение
**Выбран вариант 2 (adjacency list + рекурсивный CTE).** B2B хранит
категории таблицей с `parent_id`. Эндпоинт `GET /api/v1/categories`
отдаёт плоский список; B2C собирает дерево в
[`app/categories.py::assemble_category_tree`](app/categories.py) (O(N),
проверяет orphan по словарю id → node). `GET /api/v1/breadcrumbs?...`
проксируется в B2B, который выполняет рекурсивный CTE по `parent_id`.

## Критерии
- **Скорость запроса breadcrumbs.** Вариант 1 — лучший: один индекс,
  O(глубина) без рекурсии. Вариант 2 — рекурсивный CTE на глубину дерева
  (типично 3–5 уровней), что в категориях маркетплейса не является узким
  местом и легко кэшируется на B2C (Cache-Control: max-age=3600 для
  дерева). Вариант 3 — также быстрый, но привязан к строковому пути.
- **Сложность обнаружения orphan node.** Вариант 2 — самый явный:
  `LEFT JOIN` на стороне B2B + симметричная проверка на стороне B2C
  (узел с `parent_id`, которого нет в плоском списке, ловится в
  `assemble_category_tree` и поднимается как 422 `ORPHAN_NODE`). Вариант
  1 — нужно проверять консистентность пути и `parent_id` (две истины,
  одна должна совпадать с другой). Вариант 3 — при ошибке миграции
  путей легко получить расхождение между `parent_id` и `path`; нужно
  периодически верифицировать пары.

Adjacency list балансирует простоту схемы, инвариантную верификацию
orphan и достаточную производительность для масштаба MVP. Регрессии
проверяются тестами `test_category_tree_returns_nested_structure`
(построение дерева из плоского списка) и `test_orphan_node_returns_422`
(B2C ловит сломанную иерархию и не отдаёт частично-валидное дерево).
