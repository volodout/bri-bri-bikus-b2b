# ADR-0002: Текстовый поиск товаров

**Статус:** принято · **Дата:** 2026-05-23 · **Скоуп:** US-CAT-02

## Контекст
Покупатель ищет товары по строке. Публичный контракт B2C (`b2c.yaml`) —
`GET /api/v1/catalog/products?q=...` (`q`: `maxLength: 200`). Внутренний
контракт B2B (`b2b.yaml`) — `GET /api/v1/public/products?search=...`
(`search`: `minLength: 3`). B2C не хранит каталог — реальный матчинг по
`title` и `description` выполняет B2B. Решение в B2C сводится к трём вопросам:
(а) как публичный `q` транслируется во внутренний `search`, (б) что B2C
валидирует у себя, прежде чем дёргать B2B, (в) какой движок поиска
закладывается на стороне B2B для MVP. Здесь фиксируется третий пункт как
архитектурное ожидание B2C к контракту B2B.

## Рассмотренные подходы

1. **SQL `ILIKE '%q%'` по `title` и `description`.** Без индексов. Реализация —
   одна строка SQL, ничего не настраивается. Минусы: full table scan, нет
   ранжирования (порядок совпадений — недетерминированный), не находит
   опечатки/морфологию.

2. **`pg_trgm` + GIN-индекс, ранжирование по `similarity()`.**
   Триграммный индекс ускоряет `ILIKE`/`%`-оператор и даёт численную близость
   для сортировки. Поддерживает приближённый поиск ("iphome" найдёт "iphone").
   Минус — индекс надо создать (`CREATE INDEX CONCURRENTLY`), занимает место,
   ранжирование грубое для длинных описаний.

3. **`tsvector` / `to_tsquery` (полнотекстовый поиск Postgres) с GIN-индексом
   и `ts_rank`.** Лексический анализ, стоп-слова, морфология русского
   (`russian` dictionary), ранжирование по релевантности. Минус — самая
   сложная схема: миграция на материализованную колонку `tsvector`, выбор
   словаря, экранирование пользовательского ввода под `tsquery`.

## Решение
**Выбран вариант 2 (`pg_trgm` + GIN, `similarity()`).** B2C принимает публичный
параметр `q` и транслирует его во внутренний `search` при вызове B2B; значение
передаётся без изменений. B2B применяет триграммный поиск под условием
видимости `status = MODERATED AND deleted = false AND active_quantity > 0`.
B2C не строит свой индекс и не кеширует.

## Критерии
- **Сложность реализации на MVP.** Вариант 1 — самый простой, но без
  релевантности результаты непригодны для конверсии. Вариант 3 — заметно
  сложнее в эксплуатации и требует ручного экранирования `tsquery`.
  Вариант 2 даёт работоспособный индекс одним `CREATE EXTENSION pg_trgm` +
  одним GIN — это укладывается в спринт.
- **Релевантность результатов.** Вариант 2 ранжирует по триграммному
  сходству и устойчив к опечаткам — приоритет канона B2C-2 ("покупатель
  ушёл, если поиск возвращает мусор"). При росте каталога мы прозрачно
  переходим на вариант 3, не меняя контракт B2C (тот же `?q=...`).

## Что делает B2C (граница)
- Принимает публичный `q`, транслирует в `search` при вызове B2B.
- Валидирует длину запроса: `3 ≤ len(q) ≤ 200` (`b2c.yaml` `maxLength: 200`,
  `b2b.yaml` `minLength: 3`), иначе `400 INVALID_REQUEST`.
- Передаёт значение в B2B как есть (включая `%`, `_`, `'`) — B2C не пытается
  экранировать SQL; экранирование — обязанность B2B перед `LIKE`/`pg_trgm`.
- Совместим с фильтрами US-CAT-01: `q` живёт рядом с `category_id`,
  `filters[...]`, `sort`, `limit`, `offset`.
- Пустой результат — это `200 {items: [], total_count: 0}`, а не `404`.
