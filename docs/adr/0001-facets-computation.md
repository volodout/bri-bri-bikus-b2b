# ADR-0001: Расчёт фасетов каталога

**Статус:** принято · **Дата:** 2026-05-17 · **Скоуп:** US-CAT-01

## Контекст
B2C получает фасеты для отображения подсчётов рядом с фильтрами на странице категории.
Запрос вызывается при каждом изменении фильтров — нагрузка на источник истины (B2B)
будет существенной при большом каталоге.

## Рассмотренные подходы

1. **SQL GROUP BY на каждый запрос (B2B).** Точные счётчики на лету, всегда консистентно
   с видимостью (`status=MODERATED AND deleted=false AND active_quantity>0`).
   Минус — N агрегаций на каждый ход ползунка, линейный рост латентности и нагрузки на БД.

2. **Кэш фасетов с TTL (Redis/in-process, ключ = category_id + canonical filters).**
   Один холодный запрос, последующие — из кэша. Минус — окно неконсистентности на TTL:
   фасет может показывать товар, который только что ушёл в `BLOCKED` или `active_quantity=0`.

3. **Денормализованные счётчики в отдельной таблице,** обновляемые событиями (`product.moderated`,
   `stock.changed`). Минимальная нагрузка на чтение, но требует надёжной событийной шины и
   восстановления при дрейфе — лишняя сложность для MVP.

## Решение
**Выбран вариант 1 (SQL GROUP BY на каждый запрос).** На стороне B2C — тонкое проксирование;
B2C не кэширует фасеты в MVP. Когда нагрузка станет проблемой — слой 2 добавляется
прозрачно (ключ кэша уже формируется из category_id+filters в `app/routes/facets.py`).

## Критерии
- **Консистентность данных.** Видимость товара меняется быстро (модерация, остатки).
  Подход 1 даёт согласованность чтения с прайс-листом без окна устаревания.
- **Нагрузка на БД при большом каталоге.** В MVP объём каталога ограничен, частота
  переключений фильтров на пользователя умеренная. Когда метрика QPS на фасеты превысит
  бюджет — переходим к варианту 2 (Redis TTL ~30s), затем при необходимости — к варианту 3.
