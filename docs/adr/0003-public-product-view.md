# ADR-0003: Разделение B2B и B2C представления товара

**Статус:** принято · **Дата:** 2026-05-23 · **Скоуп:** US-CAT-03

## Контекст
`GET /api/v1/catalog/products/{product_id}` (`b2c.yaml`) возвращает покупателю
карточку товара `CatalogProductDetail`, данные для которой приходят из B2B
(`ProductPublicResponse` / `SKUPublicResponse`, X-Service-Key view). На SKU в
B2B живут приватные поля `cost_price` и `reserved_quantity` — это коммерческая
тайна продавца, в браузере покупателя их быть не должно. Любая модель/serializer
на стороне B2B может вырастить новое приватное поле — нужна архитектурно
устойчивая граница, на которой утечка не происходит «по умолчанию», даже при
ошибке разработчика B2B. Дополнительно публичный контракт `b2c.yaml`
переименовывает и вычисляет ряд полей относительно B2B, поэтому serializer
не просто фильтрует, а транслирует схему.

## Рассмотренные подходы

1. **Отдельный публичный serializer на стороне B2C с allow-list полей.**
   Чистая граница: B2C декларирует, какие поля Product и SKU видит
   покупатель; всё остальное, что прислал B2B, отбрасывается.
   Минусы — добавление нового публичного поля требует правки allow-list
   (это плюс с точки зрения безопасности, минус с точки зрения скорости
   фичи).

2. **View-level фильтрация полей с deny-list (`cost_price`,
   `reserved_quantity`).** Один проход по словарю — `pop('cost_price', None)`.
   Минусы — каждое новое приватное поле B2B нужно успеть добавить в deny-list
   до того, как оно поедет в прод; провал ревью = утечка.

3. **Отдельный публичный endpoint на стороне B2B** (например,
   `/api/v1/products/{id}/public`), который уже возвращает только разрешённые
   поля. B2C просто проксирует. Минусы — дублирование контракта между B2B
   и B2C, межсервисная координация на каждое изменение, дольше цикл фичи.

## Решение
**Выбран вариант 1 (allow-list serializer на стороне B2C).**
[`app/serializers.py`](app/serializers.py): `to_public_product` и
`to_public_sku` строят ответ поле за полем явной конструкцией словаря — приватные и
неизвестные поля не попадают в вывод by construction (allow-list сильнее
кортежа: новое поле B2B невидимо, пока его явно не добавят в билдер). Хэндлер
[`app/routes/products.py::get_product`](app/routes/products.py) гоняет
upstream-пэйлоад через этот serializer.

## Маппинг схемы B2B → `CatalogProductDetail` (`b2c.yaml`)
Serializer не только фильтрует, но и приводит схему к публичному контракту:

| B2B (`ProductPublicResponse` / `SKUPublicResponse`) | B2C (`CatalogProductDetail` / `CatalogSku`) |
|---|---|
| `title` | `name` |
| `active_quantity` | `available_quantity` |
| `price`, `discount` | `price` = `price − discount`, `old_price` = `price` при `discount > 0` иначе `null` |
| (вычисляется) | `min_price` — минимальная цена среди SKU в наличии (иначе среди всех), копейки |
| (вычисляется) | `has_stock` — есть ли SKU с `available_quantity > 0` |
| `characteristics: [{name, value}]` | `attributes: {name: value}` |
| `article` | `sku_code` |
| `status`, `seller_id`, `cost_price`, `reserved_quantity` | отброшены (нет в публичном контракте) |

Путь приведён к спецификации: `/api/v1/products/{id}` →
`/api/v1/catalog/products/{product_id}`. Вызов B2B остаётся внутренним
(`/api/v1/products/{id}`) — это граница B2B-контракта, вне скоупа US-CAT-03.

## Критерии
- **Риск утечки при добавлении нового поля в модель.** Вариант 1 — поле
  невидимо для покупателя по умолчанию, нужно явное решение его «отдать».
  Вариант 2 — поле утекает по умолчанию, спасает только дисциплина ревью
  и тесты на конкретные имена. Вариант 3 — низкий риск, но требует
  изменения B2B-контракта на каждое новое публичное поле, что замедляет
  и обходится через ошибочные расширения шире, чем нужно.
- **Сложность поддержки.** Вариант 1 — добавить поле в билдер
  `to_public_product` / `to_public_sku` в одном файле, плюс тест.
  Вариант 2 — каждый раз думать о всех приватных полях в принципе.
  Вариант 3 — синхронизация двух команд на каждое поле.

Безопасное-по-умолчанию поведение варианта 1 перевешивает скорость
варианта 2; стоимость варианта 3 не окупается на масштабе MVP.
Регрессии проверяются тестами
`test_cost_price_absent_in_response` и
`test_unknown_upstream_fields_are_dropped`.
